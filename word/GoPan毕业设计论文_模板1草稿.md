## 摘要

随着在线办公、跨终端协作和大文件共享需求的持续增长，传统本地文件管理方式在可访问性、协同效率和安全管控方面逐渐暴露出明显短板。围绕“可随时访问、可集中管理、可追溯审计”的应用目标，本文设计并实现了一个面向个人与小型团队场景的 Web 网盘系统 GoPan。系统采用前后端分离架构，后端基于 Go 语言与 go-zero 框架构建，前端基于 Vue3、TypeScript 与 Vite 实现，数据层使用 MySQL 与 Redis，并通过腾讯云对象存储（COS）完成文件对象持久化。通过该架构，系统在保证开发效率的同时兼顾了可扩展性与工程可维护性。

本文首先结合网盘系统和云存储技术相关研究，分析当前网盘在文件管理、权限控制、共享传播和数据安全方面的通用需求与关键问题；随后给出 GoPan 的总体设计方案，重点阐述用户鉴权、文件分片上传、秒传判定、目录树管理、资源分享与管理员审计等核心机制。在实现层面，系统采用 JWT 访问令牌与刷新令牌双令牌机制，结合前端拦截器实现自动续期；在上传链路中引入基于 MD5 的预上传判重逻辑，并通过分片上传提升大文件传输稳定性；在管理侧构建了数据总览、用户管理、全文件管理与审计日志等模块，支持按用户、操作类型、日期和文件扩展名进行日志检索。

最后，本文对系统进行功能测试与典型流程验证。测试结果表明，GoPan 能够稳定支持用户注册登录、文件上传下载、目录管理、资源分享保存以及管理员审计等核心业务流程，具备较好的可用性与实用性。本文工作为中小规模网盘系统的工程化落地提供了可复用的实现思路，并为后续在安全增强、性能优化与智能化管理方面的扩展奠定了基础。

## Abstract

With the rapid growth of online collaboration, cross-device access, and large-file sharing, traditional local file management can no longer satisfy modern requirements in accessibility, centralized governance, and security auditing. To address these demands, this thesis designs and implements GoPan, a web-based cloud disk system for individual users and small teams. The system adopts a front-end and back-end separated architecture. The back-end is built with Go and go-zero, while the front-end uses Vue3, TypeScript, and Vite. MySQL and Redis are used for data services, and Tencent COS is introduced for object storage. This architecture balances development efficiency, scalability, and maintainability.

This thesis first reviews existing studies on cloud disks and cloud storage, then summarizes common requirements and challenges in file management, access control, sharing, and data security. Based on this analysis, a complete system design is proposed, focusing on user authentication, chunked upload, instant upload based on file hash, directory-tree management, resource sharing, and administrative audit capabilities. In implementation, the system uses a dual-token strategy (access token + refresh token) with front-end interceptors for automatic token renewal. In the upload pipeline, MD5-based pre-upload checking is applied for deduplication, and chunked upload improves transfer reliability for large files. On the administration side, modules such as dashboard overview, user management, file governance, and audit log analysis are provided, with filtering support by actor, action type, date, and file extension.

Functional tests and typical scenario validation show that GoPan can stably support major workflows including user registration/login, file upload and download, folder management, link sharing and saving, and administrator auditing. The system demonstrates good usability and practical value. The work in this thesis provides an engineering reference for implementing medium-scale cloud disk systems and lays a foundation for future enhancements in security, performance, and intelligent operations.

**Keywords**: Cloud Disk System; Cloud Storage; Chunk Upload; JWT; Go-zero

# 第 1 章 绪论

## 1.1 研究背景与意义

随着云计算与移动互联网的普及，用户的数据生产方式已从“单机离线”转向“多端在线”。个人用户希望在手机、平板和 PC 间同步文件，团队用户希望实现快速分发与权限控制。在此背景下，网盘系统逐步从“文件备份工具”演进为“数据协作平台”。现有研究表明，基于云存储的网盘在可扩展性与可维护性方面相比传统集中式文件服务具有显著优势[1][3][5]。

另一方面，网盘系统并不只是“上传和下载”。在真实场景中，用户更关注以下能力：

1. 大文件在不稳定网络中的断点续传与上传成功率。
2. 目录层级管理、重命名与移动等“类桌面文件系统”体验。
3. 分享链接可控传播与访问后资源二次保存。
4. 管理员对用户、文件与操作行为的可观测与可审计。

因此，研究并实现一个结构清晰、功能完整、可持续演进的网盘系统，具有明确的工程意义和应用价值。本文依托 GoPan 项目展开，目标是构建一套兼顾用户体验、工程规范与安全治理的网盘系统实现方案。

## 1.2 国内外研究与应用现状

网盘系统相关研究可概括为三个方向：云存储架构、数据安全与权限治理、文件传输优化。

在国内研究方面，已有较多针对 Web 网盘和企业网盘的实现研究，重点关注分布式存储、可扩展部署和高并发访问能力[2][3][4][12]。例如，基于云存储的网盘架构研究强调分布式文件系统在扩容和容错中的价值[3]；面向企业场景的系统设计则更注重权限隔离与运维管理能力[12]。此外，国内文献也较早关注资源共享模式、共享激励与平台运营问题，这类研究补充了“系统可用”之外的“生态可持续”视角[13]。

在国外研究方面，相关工作更早进入“系统级性能优化”与“工程化稳定性验证”阶段。Gibson 等在网络附加安全磁盘研究中给出了面向文件服务扩展的系统设计思路，强调数据路径与控制路径解耦、服务扩展和吞吐提升之间的关系[10]；FAST'03 会议论文集中的相关工作则集中讨论了文件与存储系统在可靠性和性能之间的工程折中，对网盘系统中的元数据组织和存储策略具有借鉴意义[15]。在大规模分发与传输策略方面，INFOCOM 相关研究讨论了块共享与传输调度算法在复杂网络条件下的稳定性与效率问题[16]，其核心思想可迁移到网盘大文件分片上传与分发优化场景。

在安全与权限控制方面，国内外研究均认为仅依赖传统账号密码无法满足复杂系统治理需求，需结合角色权限模型（RBAC）、令牌认证和访问审计机制进行协同设计[7][8][9]。其中，JWT 机制由于无状态、易扩展，已广泛用于 Web 系统鉴权链路[8][9]。国外针对令牌存储与会话保护的研究进一步指出，浏览器端令牌存放位置、续签策略与跨站风险防护是同等重要的问题[9]，这也直接影响前后端分离系统的安全边界。

在传输性能方面，文件分片与去重是提升上传效率的重要手段。早期网络文件系统与存储研究也强调“分离元数据与数据传输链路”对系统吞吐的提升作用[10]。在网盘场景中，基于哈希判重的“秒传”机制与分片上传组合，能够在保证可靠性的同时降低重复上传成本[6]。

总体来看，国内研究在“应用落地与业务功能完整性”方面积累较多，国外研究在“底层性能、存储一致性与安全策略精细化”方面更系统。二者互补关系明显：前者强调业务可用，后者强调系统边界能力。本文在 GoPan 项目中吸收两类研究思路，一方面保证网盘基本业务闭环，另一方面在认证、上传和日志治理等关键链路中引入可验证的工程机制，实现从课程项目向工程化系统的过渡。

# 第 2 章 相关技术与理论基础

## 2.1 Go 与 go-zero 后端框架

Go 语言具备并发模型轻量、编译部署简单和工程化支持较好等优势，适合构建高并发 Web 服务。GoPan 后端采用 go-zero 框架组织 API 层、逻辑层与中间件层，形成较明确的职责边界。项目中 `handler` 负责请求解析和响应输出，`logic` 负责业务编排，`models` 负责数据访问，`middleware` 处理鉴权与管理员权限校验，具备清晰的分层结构。

从工程角度看，Go 语言在本项目中的优势主要体现为三点：

1. 部署简洁。后端服务可编译为单二进制，便于在课程环境和服务器环境快速部署。
2. 并发友好。文件上传、日志记录和管理查询等并发请求可通过 goroutine 与高性能网络库稳定处理。
3. 代码可维护。静态类型系统有利于在接口演进过程中降低回归风险。

go-zero 作为微服务友好型框架，虽然本项目以单体 API 服务运行，但其在路由管理、中间件装配和代码生成方面仍具有明显效率优势。以本项目为例，`gopan.api` 中定义的接口可直接映射到 handler 与 types，减少了手工定义 DTO 带来的重复劳动。对于毕业设计这类需要快速迭代且保持结构规范的项目，使用 go-zero 能有效平衡开发速度与代码质量。

## 2.2 Vue3 与 TypeScript 前端技术

前端采用 Vue3 组合式 API 与 TypeScript。Vue3 在组件化组织和状态复用方面具有优势，TypeScript 能在编译阶段提前发现类型错误，提升前后端联调效率。GoPan 前端通过 Pinia 管理登录态，通过 Vue Router 实现用户与管理员页面路由守卫，并在 Axios 拦截器中实现令牌注入与过期刷新，增强了客户端稳定性。

在项目落地中，Vue3 的组合式 API 使页面逻辑能够按“功能”聚合而非按“生命周期”分散。以网盘主页面为例，筛选、排序、分页、目录导航、移动对话框等能力可按职责拆分为可组合状态片段，降低组件复杂度。TypeScript 则保证了前端 `types/api` 与后端返回结构之间的一致性，避免了常见的字段拼写错误或空值处理遗漏。

前端架构中还引入了以下关键实践：

1. 鉴权前置。通过全局路由守卫在路由进入前完成登录态判断，避免页面闪烁与无效请求。
2. 接口收敛。按 `auth.ts`、`disk.ts`、`admin.ts`、`share.ts` 进行 API 模块化管理，降低跨页面耦合。
3. 状态持久化。通过 localStorage 保存 token、refreshToken 与 role，实现刷新后的状态恢复。

这类实现方式使前端不仅可用，而且具备较好的持续演进能力。

## 2.3 JWT 认证与刷新机制

JWT 在 Web 场景下可减少服务端会话存储压力，适合前后端分离系统[8][9]。GoPan 中，登录后服务端签发 `Token` 与 `RefreshToken`：

1. 常规接口携带 `Token` 访问。
2. 当返回 401 时，前端以 `RefreshToken` 请求 `/token/refresh`。
3. 刷新成功后重放原请求，失败则清空登录态并跳转登录页。

该机制兼顾了用户体验与安全控制，是系统可持续在线使用的关键基础。

在本项目中，JWT 的设计不仅用于“是否登录”的判定，还用于角色控制与接口分级。服务端在解析令牌后将用户标识、用户名和角色写入请求上下文，中间件链基于该信息继续执行管理员权限判断。该设计将“身份认证”和“权限控制”串联到同一请求链路中，避免了业务函数中重复写权限判断逻辑。

值得注意的是，JWT 机制并不天然安全。若前端存储策略或刷新策略设计不当，会带来令牌泄露与会话固定风险[9]。为降低此类风险，本项目采用短有效期访问令牌 + 较长有效期刷新令牌策略，并在刷新失败时强制清空本地凭证，避免无效会话持续存在。

## 2.4 Redis 与验证码场景

注册流程中的邮箱验证码具备“高频读写、短生命周期”特征，适合使用 Redis 缓存。GoPan 将验证码生成与校验解耦，利用 Redis 的过期机制管理验证码有效期，避免关系数据库存储临时状态带来的负担。

从数据分层角度，Redis 与 MySQL 在项目中的定位差异明确：

1. MySQL 保存结构化长期数据，如用户、目录、分享、审计记录。
2. Redis 保存短期状态数据，如验证码与可能的短时行为控制信息。

这种“冷热分离”的处理方式可降低数据库压力，并提升验证码流程的响应速度。同时，验证码有效期在 Redis 侧自动淘汰，也减少了后台定时清理任务复杂度。

## 2.5 COS 对象存储与分片上传

网盘核心数据是文件对象。若直接落盘于应用服务器，将导致容量扩展与备份成本上升。GoPan 使用腾讯云 COS 作为对象存储，并实现分片上传流程：初始化上传、分片传输、分片合并。该机制在大文件场景下可显著提升上传成功率，且便于后续扩展到断点续传策略[3][6]。

在工程实践中，将“对象存储”与“目录元数据”解耦是网盘系统的关键设计。本项目通过 `repository_pool` 与 `user_repository` 两张表完成该目标：

1. `repository_pool` 仅保存真实文件对象信息（hash、path、size 等）。
2. `user_repository` 仅保存用户视角的目录组织关系与引用信息。

这一设计带来三项收益：

1. 去重能力。不同用户保存相同文件时可复用对象，降低存储成本。
2. 共享灵活。分享保存本质上只需新增用户目录引用，不必复制对象。
3. 治理清晰。物理对象与业务目录分离后，后续回收策略可独立优化。

此外，分片上传流程采用“客户端切片 + 服务端协调 + COS 合并”的方案。相比单文件直传，该方案在不稳定网络下更具恢复能力；即使中途失败，也可定位到分片级重试，减少整文件重传概率。

## 2.6 RBAC 与审计日志

角色权限模型是后台治理的重要手段[7]。GoPan 采用普通用户与管理员双角色划分，管理员路由在后端叠加 `Auth + Admin` 中间件。与此同时，系统将登录、分享创建、分享保存等关键动作写入审计日志，为追责与统计提供依据。

RBAC 的核心价值不在于“分角色展示菜单”，而在于“将权限判断前置到统一治理层”。本项目中，用户接口统一经 `Auth` 中间件校验，管理员接口统一叠加 `Admin` 校验，从而保证权限边界在入口处生效。与在每个业务函数内部单独判断权限相比，该方式更稳定、可测试性更高。

审计日志机制则用于回答三个问题：

1. 谁在什么时候执行了什么动作。
2. 动作作用于哪个目标对象。
3. 在关键业务（分享创建、分享保存）场景下是否可追溯责任链。

在具体实现中，日志表记录 `actor_identity`、`actor_name`、`action`、`target_identity`、`detail` 等字段，并在管理员端提供按日期、扩展名、操作者等条件筛选能力。该设计不仅支持安全审计，也可为后续业务分析提供数据基础。

## 2.7 本章小结

本章围绕 GoPan 的核心技术栈与设计原理进行了系统说明。通过 Go + go-zero 构建后端服务、Vue3 + TypeScript 构建前端交互、JWT 双令牌构建认证链路、Redis 管理短时状态、COS 承载文件对象、RBAC 与审计机制保障治理能力，系统形成了较完整的技术闭环。上述技术并非孤立堆叠，而是围绕网盘业务场景协同工作，为后续需求分析、架构设计和功能实现奠定理论与工程基础。

### 3.1.3 操作可行性

用户端界面采用桌面文件管理习惯，管理员端提供聚合视图和筛选能力，学习成本较低。

从实际使用流程看，系统关键交互路径较短：用户登录后可直接进入网盘首页，核心操作按钮（上传、新建文件夹、分享、删除）均在当前页面可见；分享访问页默认展示资源信息并支持直接保存到目标目录，减少了跨页面跳转和认知负担。管理员端同样采用单页多视图结构，左侧菜单固定，右侧内容区切换，便于运维人员快速定位功能。

## 3.2 系统角色与权限边界

### 3.2.1 角色划分

系统定义两类核心角色：

1. 普通用户：管理个人网盘资源，执行上传、目录管理、分享与保存等操作。
2. 管理员：在普通用户能力之外，拥有全局统计、用户状态控制、全文件治理和日志审计能力。

### 3.2.2 权限边界原则

权限边界遵循“最小权限”原则：

1. 普通用户仅可操作自身目录树中的资源。
2. 管理员具备全局读取能力与必要治理写入能力。
3. 共享资源保存操作仅在用户登录后开放，防止匿名落库。

该原则在后端通过中间件与 SQL 条件双重落地，降低越权访问风险。

## 3.2 功能需求

### 3.2.1 普通用户功能

1. 注册登录：邮箱验证码注册、账号密码登录、令牌自动续期。

# 第 4 章 系统总体设计

## 4.1 系统架构设计

GoPan 采用 B/S 架构，前端与后端通过 REST API 交互，后端与 MySQL/Redis/COS 协作完成业务处理。总体可分为四层：

1. 表现层：Vue3 页面、表单校验、交互控制、状态管理。
2. 接口层：go-zero 路由、handler 参数绑定与响应封装。
3. 业务层：登录、上传、分享、审计、管理员治理逻辑。
4. 数据层：MySQL 持久化、Redis 缓存、COS 文件对象存储。

（TODO: 图 4-1 系统分层架构图。建议从“浏览器 -> API 服务 -> MySQL/Redis/COS”展示调用关系。）

在部署形态上，系统可采用“前后端分离 + 单后端服务 + 外部托管存储”模式：

1. 前端通过 Vite 开发代理与生产静态资源部署。
2. 后端 API 服务统一暴露业务接口。
3. MySQL 与 Redis 作为基础数据服务。
4. COS 作为文件对象服务，避免应用服务器磁盘膨胀。

该模式适合教学项目和中小规模场景，并可平滑演进到多实例部署。

## 4.2 模块设计

### 4.2.1 用户与认证模块

登录接口校验用户名与密码格式后，比对加密密码并签发令牌。注册接口结合验证码完成用户创建。认证中间件校验 `Authorization`，管理员中间件基于角色字段进行权限判定。

该模块内部可拆分为四个子能力：

1. 输入规范化：对用户名、邮箱等输入进行 trim 与格式校验。
2. 身份认证：基于 bcrypt 密码比对和用户状态检查。
3. 凭证签发：生成访问令牌和刷新令牌。
4. 权限拦截：请求进入业务逻辑前完成认证与角色判定。

### 4.2.2 文件管理模块

文件列表接口按目录 ID 分页查询，文件夹大小通过递归聚合计算。重命名、移动、删除均围绕 `user_repository` 的目录树结构进行。

目录树采用邻接表结构可以以较低复杂度支持“父子关系查找”和“分层渲染”。其代价是部分统计（如目录总大小）需要递归计算。当前实现优先保证逻辑正确性与可读性，后续可通过缓存或异步预聚合优化目录统计性能。

### 4.2.3 上传模块

预上传接口通过文件 MD5 查询公共池；命中即返回已有资源标识实现“秒传”，未命中则进入 COS 分片上传链路。上传成功后写入 `repository_pool`，再在用户目录创建引用记录，形成“物理对象与用户目录解耦”设计。

上传模块还包含前端上传状态反馈机制：

1. 显示上传进度百分比。
2. 实时计算上传速度（B/s、KB/s、MB/s）。
3. 上传完成后允许关闭状态面板。

该交互细节可明显提升用户对长耗时上传任务的可控感。

### 4.2.4 分享模块

分享创建接口生成唯一分享标识并写入 `share_link`。分享访问页可读取资源信息，并允许登录用户将资源保存到个人目录，同时记录分享保存日志。

分享模块强调“传播能力”与“治理能力”平衡：

1. 用户可快速创建可访问链接，提升资源流转效率。
2. 系统记录分享创建与保存行为，支持后续审计。
3. 分享保存不复制物理文件，减少存储冗余。

### 4.2.5 管理与审计模块

管理员端包含总览、用户、文件和日志四个子模块。审计日志表保存操作者、动作类型、目标标识与描述信息，并支持多维检索。

其中，总览模块使用聚合查询生成核心指标，日志模块支持按业务语义过滤（如文件扩展名、分享者、保存者），便于将“技术日志”转化为“可读业务记录”。

## 4.3 数据库设计

系统核心数据表包括：

1. `user_basic`：用户基础信息、状态、角色、最近登录时间。
2. `repository_pool`：公共文件池，按哈希去重。
3. `user_repository`：用户目录树与文件引用关系。
4. `share_link`：分享记录与有效期。
5. `audit_log`：关键操作审计。

（TODO: 图 4-2 数据库 E-R 图。建议突出 `user_basic` 与 `user_repository`、`share_link`、`audit_log` 的关联。）

### 4.3.1 关键字段说明

1. `user_basic.identity`：用户对外唯一标识，避免前端暴露自增主键。
2. `repository_pool.hash`：文件内容哈希，作为去重判定核心字段。
3. `user_repository.parent_id`：目录层级关系字段。
4. `share_link.expires`：分享有效期控制字段。
5. `audit_log.action`：审计动作分类字段，支持日志查询分组。

### 4.3.2 索引与约束策略

1. 用户邮箱和身份字段建立唯一索引，避免重复注册。
2. 资源哈希建立唯一索引，支撑秒传逻辑。
3. 审计日志时间字段建立索引，提升按日期筛选效率。
4. 目录层级查询字段建立组合索引，提升列表查询速度。

该策略以“查询常用路径优先”为原则，在可接受写入成本下保障读性能。

## 4.4 接口设计

系统接口分为三类：

1. 公共接口：`/user/login`、`/register`、`/code/send`、`/resource/info`。
2. 用户鉴权接口：`/file/*`、`/folder/create`、`/share/create`、`/resource/save` 等。
3. 管理员接口：`/admin/overview`、`/admin/users`、`/admin/files`、`/admin/logs`。

接口设计遵循“动词 + 资源 + 明确参数”原则，便于前后端协作和后续维护。

### 4.4.1 接口错误处理设计

接口错误处理遵循“可诊断、可提示、可恢复”原则：

1. 可诊断：后端返回明确错误文本，便于日志定位。
2. 可提示：前端将错误信息转化为用户可理解提示。
3. 可恢复：对 401 类错误触发自动刷新并重试。

### 4.4.2 典型接口设计示例

以 `/file/preupload` 为例：

1. 输入：`md5`、`name`、`ext`。
2. 输出分支 A：存在同哈希对象，返回 `identity`（秒传）。
3. 输出分支 B：不存在，返回 `key`、`upload_id`（进入分片上传）。

该设计将“判重”和“上传初始化”合并到一个前置接口，减少了前后端交互轮次。

## 4.5 安全与审计设计

### 4.5.1 输入校验设计

后端在逻辑层对用户名、密码、邮箱、文件夹名称、分享有效期等关键参数执行白名单式校验，避免非法输入直接进入持久层。

### 4.5.2 越权防护设计

除中间件鉴权外，用户级写操作均带 `user_identity` 条件进行二次约束，确保用户只能操作自身目录资源。

### 4.5.3 审计落库设计

关键动作统一通过审计写入函数落库，便于后续扩展更多动作类型。管理端查询支持多条件组合筛选，满足追踪与取证场景。

## 4.6 本章小结

本章完成了 GoPan 的总体设计，明确了系统架构、模块边界、数据模型、接口规范和安全审计机制。该设计既满足当前业务实现，也为未来性能优化和功能扩展预留了清晰接口，具备较好的工程延展性。

# 第 5 章 系统实现

## 5.1 用户认证实现

用户登录在后端完成输入规范化、用户名/密码合法性校验、bcrypt 密码比对与状态检查。登录成功后更新 `last_login_at` 并签发双令牌。前端通过 Pinia 持久化登录态，并在 Axios 响应拦截器中处理 401 自动续签，保障会话连续性。

（TODO: 图 5-1 登录时序图。建议展示“前端 -> 登录接口 -> JWT 生成 -> 本地存储 -> 路由跳转”。）

### 5.1.1 登录逻辑关键实现说明

登录逻辑可概括为以下步骤：

1. 规范化输入并校验格式（用户名字符集、密码长度）。
2. 按用户名查询用户记录并进行 bcrypt 密码比对。
3. 校验用户状态（禁用用户禁止登录）。
4. 更新最近登录时间。
5. 生成访问令牌与刷新令牌并返回。

该流程中，第 1~3 步属于安全前置校验，第 4~5 步属于会话建立阶段。分层实现后，流程职责清晰，便于后续引入登录风控策略（如失败次数限制、IP 黑名单等）。

### 5.1.2 前端令牌刷新机制实现

前端在 Axios 响应拦截器中对 401 状态进行统一处理。当首次请求失败且满足可刷新条件时，进入刷新流程：

1. 若当前已有刷新请求在执行，则将后续请求加入等待队列。
2. 刷新成功后，统一唤醒等待队列并重放请求。
3. 刷新失败则清空认证状态并提示重新登录。

该机制有效避免了“并发 401 导致重复刷新”的问题，属于前后端分离系统中的重要工程细节。

## 5.2 分片上传与秒传实现

上传模块是系统核心。

1. 客户端计算文件 MD5。
2. 调用预上传接口：若公共池已存在同哈希对象，直接返回资源标识并完成目录挂载。
3. 若不存在，调用 COS 分片初始化，按 2MB 分片逐块上传并收集 ETag。
4. 调用合并接口完成对象拼接。
5. 写入 `repository_pool` 元数据，再写入 `user_repository` 用户目录记录。

该流程显著减少了重复上传成本，并提升了大文件上传过程中的容错性。

（TODO: 图 5-2 文件上传流程图。建议标注“预上传判重”“分片循环”“合并与入库”三个阶段。）

### 5.2.1 秒传判定机制

秒传核心思想是“同内容文件只上传一次”。系统在前端计算 MD5 后调用预上传接口，后端依据哈希查询公共池：

1. 命中：返回已存在资源标识，前端仅执行目录挂载。
2. 未命中：返回分片上传参数，进入上传流程。

此机制在课程项目中有两个直接收益：

1. 显著减少重复测试上传导致的带宽与存储消耗。
2. 使“分享保存”与“二次上传”都可复用同一物理对象。

### 5.2.2 分片上传过程控制

上传组件以固定大小切分文件（当前实现为 2MB），逐片上传并记录进度。每次分片成功后更新已上传字节数，并计算平均速率。该过程保证用户可实时观测上传状态，降低大文件任务中的不确定感。

### 5.2.3 上传异常处理

上传链路的主要异常包括：

1. 前置校验失败（文件名非法、扩展名超长、文件为空）。
2. 分片请求失败（网络波动、接口错误）。
3. 合并失败或入库失败（对象参数异常、后端写入异常）。

当前版本对异常采取“终止流程 + 明确错误提示”的处理策略。后续可扩展为“分片级重试 + 断点续传”增强方案。

## 5.3 目录树与文件操作实现

系统目录采用邻接表模式（`parent_id`）构建。实现要点如下：

1. 新建文件夹时检查同目录同名冲突，并处理软删除历史重名场景。
2. 重命名时验证目标目录是否存在同名。
3. 移动时校验目标父目录存在性。
4. 删除时递归收集子节点后批量删除，保证目录完整清理。

前端通过面包屑维护当前路径，实现“根目录-子目录-返回上级”的可视化导航。

### 5.3.1 新建文件夹实现细节

新建目录时，系统先检查当前目录是否存在同名项。若发现历史软删除记录同名，会先对历史记录重命名，再执行新增，避免唯一约束冲突。该处理方式兼顾了历史数据保留与新建操作成功率。

### 5.3.2 删除递归实现细节

目录删除采用递归收集 ID 后批量删除策略。其流程为：

1. 以目标目录为根节点查询直接子项。
2. 对子目录继续递归，直到叶子节点。
3. 汇总所有节点 ID 执行批量删除。

该方式可保证删除操作的一致性，避免“父目录删除但子目录残留”的数据异常。

### 5.3.3 重命名与移动实现细节

重命名前检查目标目录同名冲突，移动前检查目标父目录存在性。由于 `file_move` 逻辑中仍有同名冲突优化 TODO，当前版本在论文中明确记录该点，体现对实现边界的真实认知。

## 5.4 分享与二次保存实现

用户可针对文件创建分享链接并设置有效期。分享页可读取资源详情，登录用户可选择目录保存。保存操作会自动处理同名冲突（追加序号），并写入 `SHARE_RESOURCE_SAVE` 审计日志，记录分享者、保存者和文件类型等信息，提升后续审计能力。

### 5.4.1 分享创建实现

分享创建接口验证资源标识和有效期后，生成全局唯一分享 ID 并落库。与此同时，系统写入 `SHARE_CREATE` 审计日志，将分享者、文件扩展名、有效期等信息编码到 detail 字段，便于后台筛选统计。

### 5.4.2 分享保存实现

分享访问页通过分享标识查询资源信息。登录用户选择目标目录后，系统执行以下逻辑：

1. 校验资源是否存在。
2. 在目标目录生成可用文件名（处理重名）。
3. 新增用户目录记录并返回新标识。
4. 写入 `SHARE_RESOURCE_SAVE` 审计日志。

该流程实现了“分享分发与用户目录管理”的平衡：既保证保存体验，又保持审计可追踪。

## 5.5 管理员端与审计实现

管理员端提供以下核心能力：

1. 总览统计：总用户、活跃用户、文件总数、总容量、当日新增。
2. 用户治理：普通用户启用/禁用。
3. 文件治理：跨用户文件检索与删除。
4. 审计检索：按动作、操作者、日期、关键字、文件类型筛选日志。

在前端页面中，通过图表和分页列表展示治理数据，使“可观测”成为系统默认能力。

### 5.5.1 数据总览实现

总览模块通过多条聚合 SQL 获取关键指标，包括：总用户、活跃用户、禁用用户、文件总数、文件夹总数、当日上传、当日注册与总文件体积。扩展名统计采用分组聚合并按数量降序输出，供前端渲染饼图和图例。

### 5.5.2 用户状态治理实现

管理员可对普通用户执行启用/禁用切换。后端会校验目标是否存在且非管理员角色，防止误操作高权限账户。状态更新成功后写入审计日志，形成治理闭环。

### 5.5.3 日志筛选实现

审计日志查询支持 `action`、`actor_name`、`file_ext`、`sharer_name`、`saver_name`、`day`、`keyword` 等复合条件。该设计使日志系统从“简单流水”升级为“可检索业务证据库”。

## 5.6 前后端联调与问题修复记录

在实现过程中，前后端联调主要遇到以下问题：

1. 令牌过期后并发请求重复刷新。
2. 大文件上传过程中的进度展示不准确。
3. 分享保存后同名冲突处理不完整。

针对上述问题，项目分别通过刷新队列机制、上传字节实时统计和可用名称生成逻辑进行了修复。该过程说明系统实现不仅包含“功能开发”，还包含“问题发现-定位-修复-验证”的完整工程环节。

## 5.7 本章小结

本章围绕认证、上传、目录管理、分享和管理审计五条核心业务链路展开实现说明，给出了关键流程与工程细节。通过分层实现与模块化联调，GoPan 已形成可稳定运行的系统版本，并具备进一步扩展的代码基础。

# 第 7 章 结论与展望

本文以 GoPan 项目为对象，完成了一个基于 Go 与 Vue3 的云存储网盘系统设计与实现。系统围绕“用户文件管理 + 管理员治理”双视角构建，形成了覆盖注册登录、文件上传、目录管理、资源分享、后台审计的完整闭环。实践表明，该系统在工程结构、功能完整性和可扩展性方面均满足毕业设计预期目标。

尽管如此，系统仍有进一步完善空间：

1. 安全增强：补充下载鉴权签名、防暴力破解策略和更细粒度权限模型。
2. 性能优化：引入异步任务队列处理重计算逻辑，优化目录递归统计开销。
3. 功能扩展：支持回收站恢复、批量操作、在线预览与协作评论。
4. 可观测性：增加指标监控、链路追踪和告警机制，提升线上可维护性。

后续工作将结合真实用户反馈持续迭代，使系统从“课程级实现”逐步演进为“生产可用型网盘服务”。
